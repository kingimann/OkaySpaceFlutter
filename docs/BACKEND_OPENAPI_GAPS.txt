      has POST /admin/badges + DELETE /admin/badges/{id}. Add a GET to list,
      or remove the app's badge-list read.

§2. OPENAPI HYGIENE — what the spec needs (so the app can stop key-probing)
================================================================================
2a. EMPTY / UNTYPED SUCCESS SCHEMAS — 57 of 487 operations declare no response
    model (200 with no/loose schema). The app must guess field names
    (e.g. token ?? access_token, data ?? items ?? results). Declare a
    response_model on each. The 33 the app actively calls (highest priority):

    POST /auth/login                 (the app key-probes the SESSION TOKEN here)
    GET  /money/requests             POST /money/request
    POST /money/requests/{id}/pay    POST /money/requests/{id}/decline
    POST /money/requests/{id}/cancel
    GET  /money/security             POST /money/security
    GET  /circles                    POST /circles            PATCH /circles/{id}
    GET  /forms/{id}                 POST /forms              POST /forms/{id}
    GET  /games                      GET  /games/{id}         POST /games
    GET  /support/tickets            GET  /support/tickets/{id}
    POST /support/tickets            POST /support/tickets/{id}/messages
    POST /support/tickets/{id}/status
    GET  /admin/ad-revenue           GET  /admin/roadside/verifications
    GET  /admin/support/tickets      POST /admin/bot/run
    GET  /promoted/account           POST /promoted/links     POST /promoted/reels
    GET  /roadside/active            GET  /roadside/helping
    POST /media/resolve-video        POST /pub/sites

2b. ERROR ENVELOPE — the spec declares ONLY 422 HTTPValidationError (471x) and
    NO schema for 4xx/5xx business errors. Today responses carry both
    {"error":{...}} and {"detail":{...}} shapes with undocumented "code"s.
    Pick ONE error envelope, declare it as the response_model on 4xx/5xx, and
    document the "code" values so the app/SDK can branch on them.

2c. AUTH-BEFORE-VALIDATION — some protected routes 422 on a missing field
    before checking the token; they should 401 first.

2d. CONSISTENCY — timestamps ISO-8601 UTC with trailing Z; IDs always strings.
